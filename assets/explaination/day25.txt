    📦 Day 25 — Role-Based Authorization

    Until now:

    ✅ Any logged-in user with a valid JWT token could access all Pet APIs.

    That meant:

    - Adopters could add pets ❌
    - Adopters could delete pets ❌
    - Shelters could add pets ✅
    - Shelters could delete pets ✅

    Although PetService already checks pet ownership before deletion, authorization should be enforced at the controller/security level before business logic runs.

    Today we added role-based authorization using Spring Security.

    =========================================================
    🎯 Goal
    =========================================================

    ADOPTER

    ✅ View pets
    ❌ Add pets
    ❌ Delete pets

    ---------------------------------------------------------

    SHELTER

    ✅ View pets
    ✅ Add pets
    ✅ Delete pets

    =========================================================
    What Was Updated?
    =========================================================

    1. Enabled Method-Level Security

    Spring Security normally protects URLs.

    To protect individual controller methods using annotations like:

    @PreAuthorize

    method security must be enabled.

    Once enabled, Spring checks user roles before allowing a controller method to execute.

    =========================================================
    2. Added Role Checks to PetController
    =========================================================

    The following endpoints were restricted:

    POST /api/pets

    DELETE /api/pets/{id}

    These endpoints now require:

    ROLE_SHELTER

    Before the controller method runs, Spring verifies that the authenticated user has the SHELTER role.

    =========================================================
    🔍 How @PreAuthorize Works
    =========================================================

    Request arrives
          ↓
    JWT validated by JwtFilter
          ↓
    User loaded from database
          ↓
    Role extracted from UserDetails
          ↓
    @PreAuthorize checks role
          ↓
    Allowed or Rejected

    If user role is:

    ROLE_SHELTER
          ↓
    Request proceeds

    If user role is:

    ROLE_ADOPTER
          ↓
    403 Forbidden

    =========================================================
    Why This Works
    =========================================================

    Your User entity already implements UserDetails.

    The getAuthorities() method converts roles into Spring Security authorities.

    Examples:

    ADOPTER
    ↓

    ROLE_ADOPTER

    ---------------------------------------------------------

    SHELTER
    ↓

    ROLE_SHELTER

    Spring Security reads these authorities automatically during authorization checks.

    =========================================================
    Security Flow After Day 25
    =========================================================

    Incoming Request
          ↓
    JwtFilter
          ↓
    Token Validation
          ↓
    User Authentication
          ↓
    Role Extraction
          ↓
    @PreAuthorize
          ↓
    Controller
          ↓
    Service Layer

    =========================================================
    Benefits of Day 25

    ✅ Shelters can add pets
